## The Problem

Mia is 28, walking through Stuttgart's Altstadt on a Tuesday lunch break.
It's 9°C and overcast. Café Müller, 80m away, has had no customers all morning.

**Neither of them knows the other exists.**

Traditional coupon apps send Mia a generic 10% off voucher — valid for 30 days —
at a restaurant she has never visited. The café stays quiet. The moment passes.

City Wallet closes that gap: **this café, this drink, right now, because the moment is right.**

## What We Built — 3 Required Modules ✅

### Module 01 — Context Sensing Layer

Aggregates real-time signals into a composite context state that triggers the generative pipeline:

| Signal | Source | Visible to User |
|---|---|---|
| Weather (temp, condition, feels like) | OpenWeatherMap API | ✅ Chip in UI |
| Location (city, district) | cityConfig.ts | ✅ Chip in UI |
| Time of day (hour, period) | Server time | ✅ Chip in UI |
| Local events (festivals, concerts) | eventsSimulator.ts | ✅ Purple chip when active |
| Merchant demand (Payone proxy) | payoneSimulator.ts | ✅ Trigger log |

**Composite context example:**
```json
{
  "weather": { "temp": 9, "condition": "overcast", "feelsLike": 6 },
  "location": { "city": "Stuttgart", "district": "Altstadt" },
  "time": { "hour": 12, "period": "lunch" },
  "merchantDemand": { "volume": 38, "status": "low" },
  "events": { "hasNearbyEvent": true, "eventName": "Stuttgarter Weindorf" }
}
```

> **Configurable by design:** All city-specific parameters live in
> `artifacts/api-server/src/config/cityConfig.ts` — swap Stuttgart for any
> city by editing one file. No codebase changes required. The merchant
> dashboard header shows: *"City: Stuttgart · Change via cityConfig.ts"*

### Module 02 — Generative Offer Engine

The merchant sets a rule. Gemini creates the offer.

**Merchant rule (set once in dashboard):**
```json
{
  "maxDiscountPercent": 20,
  "triggerStartHour": 10,
  "triggerEndHour": 14,
  "volumeThreshold": 60,
  "preferredTone": "warm"
}
```

**What Gemini 2.5 Flash generates from that rule + live context:**
> *"Cold outside? Your cappuccino is waiting."*
> Fresh batch just brewed · quiet right now · **15% off** · expires in 18:42

This is **not template-filling.** Every headline, subtext, discount percentage,
and emotional tone is generated fresh by Gemini for this user, this moment,
these exact signals. The merchant's `maxDiscountPercent` is enforced server-side:
`Math.min(parsed.discountPercent, rules.maxDiscountPercent)`.

**GDPR by design:**
- Precise location and movement data stay on-device
- Only an abstract intent signal reaches the server
- No personal data enters the Gemini prompt — only anonymized context state
- Expandable "🔒 Your privacy" panel visible in consumer UI at all times

---

### Module 03 — Seamless Checkout & Redemption

Full end-to-end verified flow:

```
Context detected → Offer generated by Gemini → Displayed to Mia
        ↓
Mia taps "Claim offer" → QR token issued (POST /api/redeem)
        ↓
Merchant scans QR → Token validated (POST /api/redeem/validate)
        ↓
Cashback credited → Redemption marked "used" (replay protection ✅)
```

Both views implemented and connected:
- **Consumer view** `/` — offer card, countdown timer, QR modal, cashback confirmation
- **Merchant dashboard** `/merchant` — live KPIs, offer performance table, rule config

## UX Design — 4 Required Points Addressed

The brief explicitly asks every submission to address these four questions.
Our answers are embedded directly in the UI via the "Why am I seeing this?" panel:

| UX Question | Our Answer |
|---|---|
| **Where does interaction happen?** | In-app card — immediate, full-attention context, no lock-screen interruption |
| **How does the offer address the user?** | Emotional-situational: *"Cold outside? Your cappuccino is waiting."* — not factual |
| **What happens in the first 3 seconds?** | Merchant + discount + expiry timer visible instantly above the fold, no scrolling needed |
| **How does the offer end?** | Accept → QR generated · Dismiss → card fades · Expiry → timer hits 0 → regenerate button |

## Privacy & GDPR

| Data | Where it stays | Reaches server? |
|---|---|---|
| Precise GPS coordinates | On-device | ❌ Never |
| Movement patterns | On-device | ❌ Never |
| Weather + time | Public API | ✅ Abstract context only |
| Intent signal | Inferred on-device | ✅ e.g. "browsing + cold weather" |
| Offer interaction | On-device | ✅ Accept / decline only |

A "🔒 Your privacy" collapsible panel in the consumer UI makes this
explicit to users at all times.

## Running Locally

**Prerequisites:** Node 20+, pnpm 9+

```bash
# Install all dependencies
pnpm install

# Set API keys
export GEMINI_API_KEY="your-gemini-key"
export OPENWEATHER_API_KEY="your-openweather-key"

# Terminal 1 — Backend
pnpm --filter @workspace/api-server run dev
# → http://localhost:8080

# Terminal 2 — Frontend
pnpm --filter @workspace/city-wallet run dev
# → http://localhost:24827
```

Consumer view: `http://localhost:24827/`
Merchant dashboard: `http://localhost:24827/merchant`

> If OpenWeather key is pending activation (new keys take up to 2 hours),
> the app falls back to `{ temp: 9, condition: "overcast", feelsLike: 6 }`
> and continues to work end-to-end. Gemini also has a fallback offer
> so the demo never shows a blank card.

## Design Decisions Worth Flagging

**In-memory store, not a database.**
A hackathon MVP doesn't need persistence — the demo restarts cleanly with
a fresh merchant and empty offers/redemptions log.

**Gemini fallback.**
If Gemini errors or returns malformed JSON, the server returns a hand-crafted
fallback offer. The demo never shows an empty card — ever.

**StrictMode-safe consumer view.**
The consumer page sources the visible offer from `useListOffers` rather than
the mutation response directly, so React 18 dev double-invocation cannot
leave the UI stuck on a skeleton loader.

**Merchant rules enforced server-side.**
`discountPercent` is clamped to `Math.min(parsed, rules.maxDiscountPercent)`
in `geminiService.ts` — the merchant's margin is always protected.

**Replay protection.**
`POST /api/redeem/validate` checks `redemption.used` before marking.
A token can only be consumed once — second attempt returns `{ valid: false }`.

## Why This Matters

The decline of inner-city retail is a structural threat to local economies
and to the Sparkassen model that DSV-Gruppe serves.

Global e-commerce has dynamic pricing, real-time demand signals, and
algorithmic personalization that local merchants simply cannot match.

City Wallet gives Café Müller the same infrastructure — privacy-first,
context-aware, and AI-powered — at zero operational cost to the merchant.

**The merchant sets a goal. The AI closes the sale.**
